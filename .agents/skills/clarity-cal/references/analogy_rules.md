# Analogy rules

Analogies are powerful because they give the user a familiar structure to think with.

They are dangerous when they are only superficially similar.

## What makes a good analogy

A good analogy should match the **relationship between parts**, not just the vibe.

Ask:

- what corresponds to what?
- what is being transferred from the familiar example to the target concept?
- does the analogy explain the mechanism, or only decorate the answer?

## Good analogy pattern

1. name the familiar system
2. map it to the target concept
3. say where the analogy breaks down

Example structure:

- “Think of memory like a library index, not the books themselves.”
- “The index helps you locate information quickly, but it isn’t the information.”
- “That breaks down because human memory also rewrites itself, while a library index usually doesn’t.”

## When to use analogies

Use them when:

- the concept is abstract
- the user lacks domain background
- a spatial or mechanical comparison will reduce confusion

Skip them when:

- the concept is already concrete enough
- the analogy would add more moving parts than it removes
- the user is asking for strict technical precision first

## Warning signs of a bad analogy

- it sounds clever but does not map cleanly
- it creates false equivalence
- it hides the important exception
- it makes the user overconfident about edge cases

## Best practice

If the analogy is doing heavy lifting, always include one sentence about where it stops being true.
