This is the <Term slug="application-factory">application factory</Term> build the senior would hand you after you finished the build-along. It is not "the answer" — it is one honest way to make every decision in the spec deliberate. Compare it against what you shipped.

## File tree

```text
kairo/
└── api/
    ├── __init__.py
    ├── app_factory.py
    └── wsgi.py
```

## `api/app_factory.py`

The factory assembles the app on purpose and returns it. Nothing runs as a side effect of import. That is the whole point — import order stops being an accident.

```python title="api/app_factory.py"
from flask import Flask, jsonify


def create_app() -> Flask:
    app = Flask(__name__)

    @app.get("/health")
    def health():
        return jsonify(status="healthy")

    return app
```

Note what is *not* here:

- No database connection, no client created on import.
- No `app.run()` hidden inside the factory.
- No module-level `app = Flask(__name__)`. The `app` only exists when somebody calls `create_app()`.

## `api/wsgi.py`

The entrypoint is the only place the app is actually constructed for a running server.

```python title="api/wsgi.py"
# api/wsgi.py
from api.app_factory import create_app

app = create_app()

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=8000)
```

The <Term slug="conventional-commits">commit is the artifact</Term> the humans read. The code is the artifact the machine reads. Keeping the factory separate from the entrypoint means a test or a migration script can call `create_app()` without booting a server.

## What the senior would change

Probably nothing on the first chapter. The spec is small on purpose. But here are the questions you should be asking yourself already, because the answers come back in later chapters:

- **Where does configuration live?** Right now, nowhere — `Flask(__name__)` uses defaults. By Chapter 2, settings come from a typed config object the factory accepts as an argument, so tests can pass a test config and production can pass a production config without touching code.
- **Where do extensions register?** They don't exist yet. By Chapter 2, the factory calls `ext_database.init_app(app)`, `ext_redis.init_app(app)`, and so on — in dependency order. The factory is where that order lives, because getting it wrong deadlocks the whole process.
- **Why `0.0.0.0` and not `localhost`?** `0.0.0.0` binds to every interface, so the container is reachable from the host. In production, a reverse proxy sits in front, so the app never sees the public interface directly.

## Self-check

Before you move on, make sure you can answer:

- Why does nothing happen on import?
- Why is `create_app()` a function and not a module-level constant?
- What would break if `app.run()` lived inside the factory?

If your build looks different and your tests pass, that is fine. The reference is one shape, not the only shape. The test suite is the contract; the reference is a second opinion.
