# The First Commit

A production system does not begin with scale. It begins with a line someone can trust.

The first engineer on a new codebase has exactly one real job: make the next step obvious. Not flashy. Obvious.

That means the first commit should do three things well:

1. create a shape other engineers can extend,
2. remove hidden setup cost, and
3. make failure visible early.

When teams skip that work, every later feature becomes slower than it should be. The system technically works, but nobody can move through it cleanly.

## Read the seam, not the slogan

This repo's M0 is a walking skeleton. That means every important seam gets touched once:

- auth,
- content loading,
- routing,
- UI primitives,
- persistence,
- tests,
- and one CLI placeholder.

That is the right first move. It is better to have one narrow path that is real than six broad paths that are pretending.

## Why idempotency shows up this early

Your first commit should be safe to repeat.

That is the practical version of **idempotency**. If a user reloads a page, retries a request, or signs in twice, the system should not drift into a weird state. It should converge on the same outcome.

In M0, progress writes should behave that way. Opening a chapter page multiple times should keep the same row current instead of spraying duplicates into the database.

## Build-along

Now build the smallest usable learning loop.

### Spec

Ship one chapter route, one protected dashboard route, and one persisted progress write. The user should be able to:

1. land on the home page,
2. sign in,
3. read the first chapter,
4. return to the dashboard,
5. and see that the app remembers something real.

### Success criteria

- the chapter renders from MDX,
- support concepts are available beside the chapter,
- the dashboard is protected,
- progress is persisted in libSQL,
- and the happy path works without manual database poking.

## Compare

The reference implementation is intentionally small. It is not trying to prove sophistication. It is proving the repo can grow without being rewritten.
